# Solution — Dataverse-side artifacts (shared)

This folder holds the **unpacked** Power Platform solution that wraps **all Code Apps in this monorepo**. The solution is shared — both apps ship together as one managed zip.

The solution unique name (in Dataverse) is **`MCSHelperCodeApps`**.

| Component | Purpose |
|---|---|
| **Workflows/Get-Agents-*.json** | Power Automate cloud flow — wraps Dataverse "List rows in environment" with FetchXML, returns agent list |
| **Workflows/Get-Transcripts-*.json** | Same pattern for conversation transcripts |
| **Other/Customizations.xml** | Connection references, root component declarations |
| **Other/Solution.xml** | Solution manifest (unique name, version, dependencies, root components) |
| **CanvasApps/msftcsa_mcsconversationviewer_6ae15.meta.xml** | Code App metadata for **MCSTranscriptViewer** |
| **CanvasApps/msftcsa_agentevaluationsviewer_dd752.meta.xml** | Code App metadata for **AgentEvalsViewer** |

> **Note:** The compiled Code App bundles (`CanvasApps/*_CodeAppPackages/`) are gitignored — they're regenerated on every `npx power-apps push` (per app) and on every `node scripts/pack-solution.mjs` (whole solution). Only the `.meta.xml` stubs are tracked.

## ⬇️ Pull cloud changes back into the repo

Made a change in the maker portal (new flow, variable, connection ref, new code app component…)? From the repo root:

```powershell
node scripts/pull-solution.mjs
```

It exports the unmanaged solution from your currently-selected `pac` env, unpacks it into `solution/src/`, strips the compiled bundles (regenerated by pack-solution), and prints a `git status` drift report. Review, commit, push.

> **Switch envs first** with `pac auth list` / `pac org select` if `pac org who` shows the wrong env.

## 🔄 Build a self-contained zip (from source)

Use this when a contributor wants to take a fresh clone and produce a working solution zip — no `npx power-apps push` required. From the repo root:

```powershell
node scripts/pack-solution.mjs            # → MCSHelperCodeApps_unmanaged_<MMDDYY_HHMM>.zip
node scripts/pack-solution.mjs --managed  # → MCSHelperCodeApps_managed_<MMDDYY_HHMM>.zip
```

What it does (see [`../scripts/pack-solution.mjs`](../scripts/pack-solution.mjs)):

1. For each app in `APPS`:
   1. `npm --prefix <app> run build` → `<app>/dist/`
   2. Mirrors `<app>/dist/` into `solution/src/CanvasApps/<appName>_CodeAppPackages/`
   3. Regenerates the `<CodeAppPackageUris>` block in that app's `.meta.xml` so it references the freshly-hashed bundle filenames
2. `pac solution pack ... --packageType Unmanaged` (or `Managed` if you pass `--managed`) → `solution/out/MCSHelperCodeApps_<type>_<MMDDYY_HHMM>.zip`

The output zip is **self-contained**: it bundles all compiled Code Apps alongside the cloud flows and the connection references. Import it into a fresh Dataverse env and the apps work end-to-end after wiring the connections + turning on the flows.

> **Note:** `pack-solution` rewrites each `.meta.xml`'s `<CodeAppPackageUris>` block to match the current build's content-hashed filenames. Expect `.meta.xml` diffs after each pack — that's normal.

### Manual pack (if you don't want to rebuild)

```powershell
pac solution pack `
  --folder solution\src `
  --zipFile solution\out\MCSHelperCodeApps.zip `
  --packageType Unmanaged
```

Then import via maker portal *or*:

```powershell
pac solution import --path solution\out\MCSHelperCodeApps.zip
```

### Unpack (after re-exporting from maker → refresh local source)

```powershell
pac solution unpack `
  --zipFile <path-to-downloaded-export>.zip `
  --folder solution\src `
  --packageType Unmanaged
```

> ⚠️ Unpack overwrites files in-place. Commit local changes first. The `pull-solution.mjs` script does this for you with `--allowDelete` to mirror cloud deletions.

## 🚀 Releases

Releases are produced by [`.github/workflows/release.yml`](../.github/workflows/release.yml), which:

1. Reads `<Version>` from `solution/src/Other/Solution.xml`
2. `pac solution export --managed` from the target Dataverse env (controlled by `PAC_ENVIRONMENT_ID` secret)
3. Uploads the managed zip to a GitHub Release tagged `v${version}`

The release ships **whatever is currently published in the target Dataverse env** — not what's in source. If you want source to be the release artifact instead, switch the workflow to call `node scripts/pack-solution.mjs --managed` (deferred — see [`workflows-disabled/README.md`](../.github/workflows-disabled/README.md)).

Required repo secrets: `PAC_APP_ID`, `PAC_CLIENT_SECRET`, `PAC_TENANT_ID`, `PAC_ENVIRONMENT_ID`.

## 🔌 Flow input legend (verified against trigger schema)

Both flows use the same shape — Logic Apps trigger field IDs `text` / `text_6` map to the human-readable titles set in the Power Automate UI:

| Field key | UI title | Meaning |
|---|---|---|
| `text` | `envUrl` | Dataverse environment URL (e.g. `https://orgname.crm.dynamics.com`) |
| `text_6` | `fetchXml` | FetchXML query string |

The codegen for `MCSTranscriptViewer/src/generated/services/Get_AgentsService.ts` and `Get_TranscriptsService.ts` uses the *keys* (`text` / `text_6`), not the titles — that's a Power Apps quirk. The full legend lives in `MCSTranscriptViewer/src/components/BrowseFlows/flowDataSource.ts` for any code-side consumers.

## 🆕 First-time env setup (new contributor)

1. From repo root: `node scripts/pack-solution.mjs` — produces `solution/out/MCSHelperCodeApps_unmanaged_<stamp>.zip`
2. `pac solution import --path solution\out\MCSHelperCodeApps_unmanaged_<stamp>.zip`
3. In maker → **Connection References**, set the Dataverse, Copilot Studio, and O365 Users connections
4. In maker → **Cloud flows**, **turn on** both `Get-Agents` and `Get-Transcripts`
5. (Only if you'll iterate on an app locally) `cd <app>` → `npx power-apps refresh-data-source` to regenerate `src/generated/` against your env, then `npm run dev` / `npx power-apps push`

See the root [`README.md`](../README.md) for ongoing dev workflow.
